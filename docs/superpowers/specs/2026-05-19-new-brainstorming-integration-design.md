# EM-SKILL /em new 命令 + brainstorming 整合设计

**日期**: 2026-05-19
**版本**: v1.0
**状态**: 已批准

## 1. 概述

将 `/em new` 命令与 brainstorming 工作方式整合，实现嵌入式场景下的需求驱动设计流程。

### 触发方式
```
/em new <功能描述>
```

## 2. 流程概览

```
┌─────────────────────────────────────────────────────────────┐
│  /em new <功能描述>                                         │
│       ↓                                                    │
│  1️⃣ 【追问澄清】（全面维度）                               │
│       ↓                                                    │
│  2️⃣ 【方案提案】                                           │
│       ↓                                                    │
│  3️⃣ 【设计确认】                                           │
│       ↓                                                    │
│  4️⃣ 【文档化】→ .emv2/discussion/<需求ID>/                 │
│       ↓                                                    │
│  5️⃣ 【更新 project-spec】                                   │
└─────────────────────────────────────────────────────────────┘
```

## 3. 文件结构

```
.emv2/discussion/<需求ID>/
├── discussion.md      # 讨论记录（合并原5个文件）
└── design.md          # 设计文档
```

**替代原文件**：
- split.md, requirements.md, hardware.md, brainstorm.md → discussion.md
- milestones.md → design.md

## 4. 对话风格：混合模式

### 步骤 1: 接收需求

```
用户: /em new 我想加一个CAN通信模块

AI: 收到，正在分析需求...
```

### 步骤 2: 追问澄清（一次一问，全面维度）

| 维度 | 问题示例 |
|------|---------|
| **硬件** | "需要用哪组 CAN 外设？引脚怎么分配？" |
| **协议** | "波特率多少？帧格式是什么？" |
| **实时性** | "消息处理用中断还是轮询？有 deadline 吗？" |
| **资源** | "预估缓冲区需要多大？" |
| **兼容性** | "和现有模块会冲突吗？" |
| **验证** | "怎么验证功能？串口输出还是 LED？" |
| **异常** | "错误怎么处理？超时设多少？" |

### 步骤 3: 方案提案

**复杂度判断**：

| 分类 | 判断条件 | 方案数量 |
|------|---------|---------|
| **简单** | ≤2行，无多外设/协议/状态机 | 单一方案 |
| **中等** | 1-2个外设或协议 | 2 方案 |
| **复杂** | 多外设/多协议/状态机 | 3 方案 |

**输出格式**：
```markdown
【方案 A】<名称> [推荐]
  - 优点：<列举>
  - 缺点：<列举>
  - 适用：<场景>

请选择 [A/B/C] 或提出其他想法。
```

### 步骤 4: 设计确认

```
【设计确认】

方案: <方案名称>
核心组件:
  - <组件1>
  - <组件2>

  [1] 确认  [2] 修改  [3] 重新提案
```

```markdown
- 响应时间

- 中断数量

## 7. 验证方案

- 预期结果

- 错误处理
- 恢复机制

---
批准人: <用户>
批准日期: <日期>
```

## 6. 更新 project-spec

设计文档完成后：
1. 读取 `.emv2/project-spec.md`
2. 提取最大 S 编号 +1
3. 添加新功能模块（带设计文档引用）
4. 更新步骤表
5. 更新 memory-log.md

## 7. 硬性门控

⚠️ **用户批准设计前，禁止执行**：
- 编写任何代码
- 创建/修改源文件
- 执行编译/烧录命令

## 8. 与 disc 的关系

| 命令 | 用途 |
|------|------|
| `/em new` | 主入口 + brainstorming（重构） |
| `/em disc` | 快速讨论（保留） |
