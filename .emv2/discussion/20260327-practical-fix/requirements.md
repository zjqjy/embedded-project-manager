# 需求讨论 - 实用性修复

**讨论日期**: 2026-03-27
**讨论ID**: 20260327-practical-fix
**参与方**: 用户 + AI

---

## 用户原始需求

```
当前版本存在以下问题：
1. 芯片的选择有点少，而且麻烦
2. 新功能开发前，没有进入讨论阶段，让需求对齐，以及头脑风暴
3. 人机验证阶段：工作流不太明确，当前用户提交一个失败记录后不会自动触发问题记录，也没有进入讨论阶段和人类一起交流沟通，然后一个失败记录后面解决了之后也没有记录
4. 随着工程量的加大，两个文件会越来越大，会导致模型注意力不集中
5. 嵌入式项目调试一般是串口打印，要求本技能生成的代码要带调试信息，最好要是单片机代码运行结束之后可以自动读取打印串口信息（可能要开发一个工具？）
6. 效率问题，当前是线性开发，有什么合适的流程改善这个过程呢
```

---

## 讨论结论

### 问题1: 芯片选择
- **方案**: 初始化选项 + 学习用户常用芯片
- **触发时机**: 存量接入时 + 项目初始化时
- **存储位置**: 全局 `~/.claude/chips.json`
- **学习粒度**: 芯片型号

### 问题2: 需求对齐讨论流程 🔴 P0
- **讨论形式**: 明确选项 + 不明确AI引导提问
- **并行程度**: 必须讨论完才能规划
- **大需求拆分**: 拆为 S[X]-子功能 形式
- **讨论阶段**:
  1. 需求讨论 → requirements.md
  2. 规则讨论 → rules.md
  3. 硬件对齐 → hardware.md
  4. 子流程拆分 → milestones.md

### 问题3: HVR工作流增强 🔴 P0
- **失败触发**: 自动记录问题 → 触发讨论
- **闭环记录**: 成功后更新"已解决问题"
- **讨论整合**: 问题分析与人类协作

### 问题4: 文件归档机制
- **归档路径**: `.emv2/history/<年份>/<月份>/<日期>/`
- **触发方式**:
  - 会话结束自动归档
  - 文件过大自动归档（>100行）
  - 手动命令: `归档`
- **引用格式**: `[.emv2/history/<年>/<月>/<日>/problem-log.md#标题]`

### 问题5: 串口调试工具 🔴 P0
- **工具类型**: GUI (tkinter)
- **日志用途**: B - 自动发给AI分析
- **配置方式**: JSON配置文件
- **日志存储**: `.emv2/logs/serial_<步骤>_<问题>.log`
- **触发方式**: 手动按钮 + 自动触发（检测关键日志）
- **用户参与**: 人工烧录，工具记录，AI读取分析

### 问题6: 效率问题
- **解决方案**: S2-S6 规划中的各项改进
- **优先级**: P0优先开发 S2、S3、S4
