## 实用性修复规划（2026-03-27 讨论确定）

### S2: 需求对齐讨论流程 🔴 P0优先
- **状态**: ✅ 规划完成
- **前置条件**: 无
- **功能描述**: 强制进入需求讨论模式，包含硬件对齐、需求澄清、规则确认、大需求拆分
- **讨论形式**: 明确选项 + 不明确AI引导提问
- **前置要求**: 必须讨论完才能规划
- **复杂度评估机制**: 根据功能复杂度自动选择流程
  - 简单 + 无新增硬件 → 精简流程A（硬件确认 → 直接开发）
  - 简单 + 少量新增 → 精简流程B（硬件对齐 → 需求确认）
  - 中等复杂度 → 标准流程（硬件 → 需求 → 规则 → 里程碑）
  - 复杂/高风险 → 完整流程（全部4阶段 + 额外分析）
- **讨论阶段顺序**（已调整）:
  1. **硬件对齐**（最先）- 引脚、电平、操作方式
  2. **需求讨论** - 功能目标、数据处理、输出目标
  3. **规则讨论** - 约束条件、边界情况
  4. **子流程拆分** - 大需求拆为S[X]-子功能
- **开发内容**:
  1. **复杂度评估模块**
     - 功能复杂度评估（简单/中等/复杂）
     - 硬件影响评估（无/少量/较多新增）
     - 风险程度评估（低/中/高风险）
     - 自动选择对应流程
  2. **精简流程模块**
     - 精简流程A：快速确认 → 直接开发
     - 精简流程B：硬件确认 → 需求确认 → 开发
  3. **标准流程模块**
     - 硬件对齐模块（最先）- 引脚、电平、冲突检查、芯片手册确认
     - 需求讨论模块 - 基于硬件向上追溯
     - 规则讨论模块 - 约束条件、边界情况
     - 子流程拆分 - S[X]-子功能 形式
  4. **完整流程模块**
     - 标准流程 + 额外分析
     - 详细风险评估
     - 多轮讨论支持
- **验证方式**: 用一个实际需求测试完整讨论流程

### S4: 芯片学习机制 🔴 P0优先
- **状态**: 🚧 开发中
- **前置条件**: S2 完成
- **功能描述**: 存量接入和初始化时学习用户常用芯片

### S5: 串口调试工具
- **状态**: 🚧 开发中
- **前置条件**: S3 完成
- **功能描述**: AI调用后台串口监听工具，支持GUI配置和独立关闭
- **开发内容**:
  1. **GUI界面 (tkinter)** ✅
     - 串口配置（端口、波特率、数据位、校验位、停止位）
     - JSON配置文件加载
     - 实时日志显示
     - 关闭窗口不停止监听
  2. **日志管理** ✅
     - 存储到 `.emv2/logs/serial_{{STEP}}_{{PROBLEM}}.log`
     - 自动解析特定格式日志
     - 关键日志标记（[DBG]/[ERR]/[PASS]/[FAIL]）
  3. **触发机制** ✅
     - AI调用启动（`开始验证-步骤xx`）
     - 结束: `结束监听` 或 `DEBUG_TEST_END` 标记
  4. **AI接口** ✅
     - MCP调用: start/stop/status/notify
     - 状态文件共享: `.emv2/logs/.serial_status.json`
  5. **单片机日志参考代码** ✅
     - templates/serial-log-reference.md
     - 支持STM32/GD32/CH32系列
- **验证方式**: 在实际项目中测试烧录+记录+AI分析流程

  5. **归档索引** ✅
     - 模板: templates/history-index.md
