# 嵌入式项目开发管家 v3

## 核心功能

1. **存量项目接入** - 分析现有代码，重建项目状态
2. **需求对齐讨论** - 新功能开发前必须进行需求、规则、硬件对齐讨论
3. **人机协作测试** - AI准备测试，人工执行硬件验证
4. **上下文管理** - 无缝衔接多个开发会话，全局项目索引

---

## 文件组织结构

```
<用户项目根目录>/
│
├── User/                          # 用户代码（原有内容）
│
└── .emv2/                         # 嵌入式项目管家生成（所有工具文件）
    ├── project-spec.md            # 项目规格单
    ├── memory-log.md              # 记忆日志
    ├── problem-log.md            # 问题追踪
    ├── decision-log.md           # 决策记录
    │
    ├── discussion/               # 讨论结果
    │   └── <需求ID>/
    │       ├── requirements.md   # 需求讨论结果
    │       ├── rules.md          # 规则讨论结果
    │       ├── hardware.md       # 硬件对齐结果
    │       └── milestones.md     # 子流程拆分结果
    │
    ├── history/                   # 归档
    │   └── index.md              # 归档索引
    │
    ├── checkpoints/              # 会话检查点
    │   └── session-<id>.json
    │
    └── logs/                      # 串口日志
        └── serial_<步骤>_<问题>.log
```

**注意**: `.emv2/` 目录隔离用户代码和工具文件，建议加入 `.gitignore`

**文件说明**:
- `memory-log.md` - 当前项目状态（步骤、待办、会话历史）
- `project-spec.md` - 项目规格和开发规划
- AI 需要时从这两个文件动态生成摘要，无需额外文件

## 命令使用指南

### 1. 存量接入

**触发**: 用户输入 `存量接入 <路径>`

**执行流程**:
```
1. 进入指定目录
2. 分析代码结构：
   - 识别芯片型号（查找 system_xxx.c, startup_xxx.s）
   - 识别已初始化外设（gpio, usart, i2c, spi, timer等）
   - 识别驱动层（已驱动的传感器/设备）
   - 识别应用层（主功能循环）
   - 识别调试接口（printf重定向，LED指示）
   - 检查git状态（最后一次提交信息）
3. 【芯片学习】
   - 检查 ~/.claude/chips.json 是否存在
     - 不存在 → 自动创建: {"chips": [], "builtin": [内置芯片列表]}
   - 检测芯片型号：使用正则提取 system_xxx.c / startup_xxx.s
   - 识别厂商：ST → ST, GD → GigaDevice, CH32/WCH → WCH
   - 检测外设：搜索 gpio_init, usart_init, i2c_init 等关键词
   - 添加到 chips.json 的 chips 数组（source: "learned"）
   - 如芯片已存在 → 更新 lastUsed 时间戳
4. 创建 .emv2/ 目录结构
5. 生成/更新 .emv2/project-spec.md 和 .emv2/memory-log.md
6. 更新全局索引 ~/.claude/embedded-projects-index.md
7. 输出简单审计报告
```

**输出格式**:
```
🔍 代码审计完成

识别结果:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
📟 芯片型号: [从代码识别]
✅ 已完成:
   - [步骤]: [描述]

⚠️ 部分完成:
   - [步骤]: [描述]

🔄 当前状态推测:
   - 步骤: [当前步骤]
   - 问题: [发现的问题]
   - 上次提交: [git log信息]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

已生成项目文档:
- .emv2/project-spec.md
- .emv2/memory-log.md
- 已更新全局索引 ~/.claude/embedded-projects-index.md
```

### 2. 新功能开发（讨论前置）

**触发**: 用户输入 `新功能开发 <功能描述>`

**重要**: 新功能开发**必须先进行需求对齐讨论**，不能跳过。

**执行流程**:
```
1. 读取当前项目的 .emv2/project-spec.md
2. 分析新功能需求
3. 生成需求ID（格式: <日期>-<简短描述>）
4. 【自动进入】需求对齐讨论流程（不需额外命令）
5. 讨论完成后，评估对现有代码的影响
6. 规划增量开发步骤
7. 更新 .emv2/project-spec.md（添加新功能模块）
8. 更新 .emv2/memory-log.md（记录新功能规划）
9. 更新全局索引
```

**输出格式**:
```
🆕 新功能开发

功能描述: [用户输入的功能描述]
讨论ID: [需求ID]

━━━━━━━━━━━━━━━━━━━━━━━━━━
📋 需求对齐讨论（自动进入）

讨论ID: <需求ID>
当前阶段: 1/4 (需求讨论)
...
```

### 3. 开始讨论（需求对齐讨论流程）

**触发**: 用户输入 `开始讨论` 或 `开始讨论 <讨论ID>`

**执行流程**:
```
阶段1: 需求拆分
1. AI分析需求涉及的子系统/模块
2. 用户确认/修改拆分结果
3. 讨论结果保存到 .emv2/discussion/<需求ID>/split.md

阶段2: 需求确认
1. 对每个子功能确认详细参数（通信/测量/控制/状态机）
2. 按类型问相应问题
3. 用户确认后直接保存到 .emv2/discussion/<需求ID>/requirements.md

阶段3: 硬件对齐
1. 对涉及硬件的子功能确认引脚和配置
2. 检查引脚冲突
3. 用户确认后保存到 .emv2/discussion/<需求ID>/hardware.md

阶段4: 头脑风暴
1. 讨论技术难点和解决方案
2. 用户确认解决方案
3. 保存到 .emv2/discussion/<需求ID>/brainstorm.md

阶段5: 子流程拆分
1. 拆分为 S[X]-子功能 形式
2. 确定验证方式
3. 保存到 .emv2/discussion/<需求ID>/milestones.md
4. 更新 .emv2/project-spec.md

```

**讨论形式**:

- **AI引导提问**: AI通过提问引导用户思考不明确的需求
- **用户追问**: 用户可随时提出问题或补充信息
- **确认即保存**: 用户确认后AI直接保存，不需要二次确认

**输出格式**:
```

讨论ID: <需求ID>

- 简述：[一句话描述]
```

## 新流程概述

**触发**: 用户输入 `新功能开发 <名称>`

**完整流程**（中等复杂度以上）：
```
1. 用户输入: 新功能开发 <名称>
   ↓
2. 【需求拆分】
   AI分析需求 → 输出涉及的子系统/模块列表
   ↓
3. 【需求确认】（对每个子功能）
   逐项确认参数、接口、协议
   ↓
4. 【硬件对齐】（对每个涉及硬件的子功能）
   引脚分配、电平、通信协议
   ↓
5. 【头脑风暴】（都要，简单功能除外）
   技术难点 + 解决方案
   ↓
6. 【子流程拆分】
   拆分为 S[X]-子功能 形式
   ↓
7. 【保存文件】
   - split.md（需求拆分）
   - requirements.md（需求确认）
   - hardware.md（硬件对齐）
   - brainstorm.md（头脑风暴）
   - milestones.md（子流程拆分）
```

**简化流程**（特别简单的功能）：
- 需求确认 → 直接开发
- 或：需求确认 → 硬件确认 → 开发

### 阶段1: 需求拆分

**目标**: 分析需求涉及的子系统/模块

**触发时机**: 讨论开始时

**AI行为**:
1. 读取用户需求描述
2. 分析涉及哪些硬件/子系统/控制器
3. 列出子系统列表供用户确认
4. 【确认后保存到 split.md】

**输出格式**:
```
## 需求拆分

根据需求描述，分析出以下子系统/模块：

### 1. [子系统A名称]
- 类型：通信/测量/控制/显示/其他
- 简述：[一句话描述功能]

### 2. [子系统B名称]
- 类型：通信/测量/控制/显示/其他
- 简述：[一句话描述功能]

### 3. [其他]
- ...

---
请确认/修改这个拆分结果。
输入 `确认` 继续，或补充/修改子系统。
```

**拆分维度**:
| 类型 | 说明 | 示例 |
|------|------|------|
| 通信 | 数据交互 | CAN/UART/I2C/SPI |
| 测量 | 信号采集 | ADC/PWM测量/编码器 |
| 控制 | 输出驱动 | GPIO/PWM/继电器 |
| 显示 | 人机交互 | LED/LCD/串口打印 |
| 状态机 | 流程管理 | 多状态切换 |
| 存储 | 数据管理 | Flash/EEPROM |

**拆分原则**:
- 每个子系统应可独立开发和测试
- 记录子系统间的依赖关系
- 区分"已有"和"新增"的子系统

**完成后**: AI确认用户输入后 → 保存到 `.emv2/discussion/<需求ID>/split.md`

1. 依次处理每个子功能
2. 针对子功能类型，问相应的问题
3. 【明确参数后直接保存，不等二次确认】

**子功能类型对应问题**:
根据子系统类型选择相应的问题模板（不需要全部使用）：

```
  - 地址/ID：
  - 帧格式：
**数据内容**:
  - 发送什么：
  - 接收什么：
**实时性要求**: 实时 / 非实时 / 有deadline

请补充或确认...
```

#### 测量类（如涉及测量）
```
### [子系统名称] - 需求确认

**测量对象**: （测什么信号）
**测量精度**: （精度要求）
**测量方式**:

**数据处理**:

```

#### 控制类（如涉及控制）
```
### [子系统名称] - 需求确认

**控制目标**: （控制什么）
**控制方式**:
  1. GPIO开关
  2. PWM输出
  3. DAC输出
**参数**:
  - 频率/占空比：
  - 通道：
**触发方式**:
  1. 定时触发
  2. 外部触发
  3. 软件触发

请补充或确认...
```

#### 状态机类（如涉及状态机）
```
### [子系统名称] - 需求确认

**状态列表**:
  1. [状态A] - 描述
  2. [状态B] - 描述

**状态转换条件**:
```

### 阶段3: 硬件对齐

**目标**: 对涉及硬件的子功能，确认引脚和硬件配置

**触发时机**: 需求确认完成后

**AI行为**:
1. 对每个涉及硬件的子功能
2. 读取 project-spec.md 中的已有硬件配置
3. 检查冲突，确认引脚分配
4. 【确认后保存到 hardware.md】

**输出格式**:
```
### [子系统名称] - 硬件对齐

**已有硬件**:
- [列出已有的相关外设和引脚]

**新增硬件需求**:
1. **IO类型**:
   1. 数字输入（DI）
   2. 数字输出（DO）
   3. 模拟输入（AI）
   4. 通信接口

2. **引脚分配**: _________

3. **电平**: 1. 3.3V  2. 5V  3. 其他

4. **复用功能**: （如有）

5. **中断需求**:
   1. 不需要
   2. 边沿触发
   3. 电平触发

6. **DMA需求**:
   1. 不需要
   2. 需要（用于什么）

7. **芯片手册**: 1. 有  2. 无

---
如有冲突或疑问，请提出。
```

**引脚冲突检查**:
```
AI自动检查：
1. 复用功能冲突
2. 已有GPIO占用
3. 外设通道冲突
4. 中断优先级冲突
```

**完成后**: 用户确认 → 保存到 `.emv2/discussion/<需求ID>/hardware.md`

### 阶段4: 头脑风暴

**目标**: 讨论技术难点和解决方案

**触发条件**: 所有中等复杂度以上功能都要进入此阶段

**触发时机**: 硬件对齐完成后

**AI行为**:
1. 基于前几个阶段的讨论结果
2. 识别技术难点
3. 提出可能的解决方案
4. 用户确认或提出其他方案
5. 【确认后保存到 brainstorm.md】

**输出格式**:
```
## 头脑风暴

基于前面的讨论，以下是识别的技术难点和解决方案：

### 难点1: [描述]
**分析**: [AI分析原因]
**可能方案**:
  A. [方案A] - 优点/缺点
  B. [方案B] - 优点/缺点
**建议**: [AI推荐]
**确认**: [请用户选择或补充]

### 难点2: [描述]
...

---
请确认各难点的解决方案。
输入 `继续` 进入下一阶段。
```

**头脑风暴内容**:
| 类型 | 内容 |
|------|------|
| 时序问题 | 中断延迟、响应时间 |
| 资源冲突 | 外设复用、优先级 |
| 可靠性 | 错误处理、恢复机制 |
| 性能 | 实时性优化、算法效率 |
| 兼容性 | 协议一致性、版本兼容 |

**完成后**: 用户确认 → 保存到 `.emv2/discussion/<需求ID>/brainstorm.md`

### 阶段5: 子流程拆分

**目标**: 将大需求拆分为可独立开发的子功能

**触发时机**: 头脑风暴完成后

**AI行为**:
1. 基于前几个阶段的讨论结果
2. 按依赖关系排序子功能
3. 确定每个子功能的验证方式
4. 【确认后保存到 milestones.md】

**输出格式**:
```
## 子流程拆分

基于讨论结果，建议拆分为以下子流程：

### S[X]-A: [子功能A名称]
- **所属子系统**: [哪个]
- **开发内容**:
  1. [任务1]
  2. [任务2]
- **前置条件**: 无
- **验证方式**: [如何验证]
- **优先级**: 1

### S[X]-B: [子功能B名称]
- **所属子系统**: [哪个]
- **开发内容**:
  1. [任务1]
- **前置条件**: S[X]-A 完成
- **验证方式**: [如何验证]
- **优先级**: 2

---

请确认：
1. 拆分是否合理？
2. 优先级是否正确？
3. 验证方式是否可行？
```

**拆分原则**:
- 每个子功能应可独立验证
- 有依赖关系的子功能需明确前置条件
- 验证方式应具体可执行

**完成后**: 用户确认 → 保存到 `.emv2/discussion/<需求ID>/milestones.md`

### 讨论状态管理

**状态文件**: `.emv2/discussion/<需求ID>/status.json`
```json
{
  "discussion_id": "<需求ID>",
  "created_at": "<创建时间>",
  "current_phase": 1,
  "phases": {
    "split": "completed",
    "requirements": "in_progress",
    "hardware": "pending",
    "brainstorm": "pending",
    "milestones": "pending"
  },
  "subsystems": ["子系统A", "子系统B"],
  "current_subsystem": 0
}
```

**状态转换**:
```
用户输入 "继续" → 保存当前结果 → 进入下一项/下一阶段
用户输入 "跳过" → 跳过当前子功能 → 进入下一个
用户输入 "退出" → 保存进度 → 退出讨论
用户输入 "继续讨论" → 从上次位置继续
```

### 10. 串口调试工具

**触发**: AI自动调用（用户输入 `开始验证-步骤xx` 时）

**工具位置**: `scripts/serial_tool.py`

**配置文件**: `templates/serial_config.json`

**执行流程**:
```
用户: 开始验证-步骤xx
   ↓
AI生成HVR文件
   ↓
AI调用: python serial_tool.py start <config_json>
   ↓
工具后台启动，有GUI显示状态
   ↓
工具检测到串口数据 → 写入日志文件
   ↓
AI可调用: python serial_tool.py notify 检查新数据
   ↓
结束条件（二选一）:
   - 用户输入: `结束监听`
   - 程序输出: DEBUG_TEST_END 标记
   ↓
AI调用: python serial_tool.py stop
```

**MCP调用接口**:
```bash
# 启动监听
python scripts/serial_tool.py start templates/serial_config.json

# 检查新数据通知
python scripts/serial_tool.py notify

# 获取状态
python scripts/serial_tool.py status

# 停止监听
python scripts/serial_tool.py stop
```

**配置文件格式** (templates/serial_config.json):
```json
{
  "serial": {
    "port": "COM5",
    "baudrate": 115200,
    "bytesize": 8,
    "parity": "N",
    "stopbits": 1,
    "timeout": 1
  },
  "end_marker": "DEBUG_TEST_END",
  "log_file": ".emv2/logs/serial_{{STEP}}_{{PROBLEM}}.log",
  "keywords": {
    "debug": "[DBG]",
    "error": "[ERR]",
    "pass": "[PASS]",
    "fail": "[FAIL]"
  }
}
```

**GUI功能**:
- 显示监听状态（运行中/已停止）
- 实时显示接收到的日志内容
- 串口配置修改（端口、波特率、数据位、校验位、停止位）
- 关闭窗口不停止监听（后台继续运行）

**AI通知机制**:
- 工具通过写入 `.emv2/logs/.serial_status.json` 共享状态
- AI调用 `notify` 命令检查是否有新数据
- 状态文件包含: status, last_keyword, last_line, has_new_data

**单片机日志参考代码**:
- 模板: templates/serial-log-reference.md
- 统一日志格式: [DBG]/[ERR]/[PASS]/[FAIL]/DEBUG_TEST_END
- 支持STM32/GD32/CH32系列

### 外设自动检测逻辑

**检测方法**: 在代码中搜索外设初始化函数调用

| 外设 | 检测关键词 | 典型函数 |
|------|-----------|----------|
| GPIO | `gpio_init`, `GPIO_MODE` | `gpio_init(GPIOx, ...)` |
| UART/USART | `usart_init`, `uart_init` | `usart_init(USARTx, ...)` |
| I2C | `i2c_init` | `i2c_init(I2Cx, ...)` |
| SPI | `spi_init` | `spi_init(SPIx, ...)` |
| CAN | `can_init` | `can_init(CANx, ...)` |
| Timer | `timer_init` | `timer_init(TIMERx, ...)` |
| ADC | `adc_init` | `adc_init(ADCx, ...)` |
| DAC | `dac_init` | `dac_init(DACx, ...)` |
| USB | `usb_init` | `usb_init(USBx, ...)` |

**检测流程**:
```
1. 扫描所有 .c 文件
2. 搜索外设关键词
3. 提取_unique_外设列表
4. 添加到芯片信息的 peripherals 字段
```
