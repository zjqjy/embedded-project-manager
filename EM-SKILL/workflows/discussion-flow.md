# 讨论流程细则

## 完整流程（中等复杂度以上）

```
1. 用户输入: /em new <名称>
   ↓
2. 【需求拆分】
   AI分析需求 → 输出涉及的子系统/模块列表
   ↓
3. 【需求确认】（对每个子功能）
   逐项确认参数、接口、协议
   ↓
4. 【硬件对齐】（对每个涉及硬件的子功能）
   引脚分配、电平、通信协议
   ↓
5. 【头脑风暴】（都要，简单功能除外）
   技术难点 + 解决方案
   ↓
6. 【子流程拆分】
   拆分为 S[X]-子功能 形式
   ↓
7. 【保存文件】
   - split.md（需求拆分）
   - requirements.md（需求确认）
   - hardware.md（硬件对齐）
   - brainstorm.md（头脑风暴）
   - milestones.md（子流程拆分）
```

## 简化流程（特别简单的功能）

- 需求确认 → 直接开发
- 或：需求确认 → 硬件确认 → 开发

---

## 阶段1: 需求拆分

**目标**: 分析需求涉及的子系统/模块

**输出格式**:

```
## 需求拆分

根据需求描述，分析出以下子系统/模块：

### 1. [子系统A名称]
- 类型：通信/测量/控制/显示/其他
- 简述：[一句话描述功能]

### 2. [子系统B名称]
- 类型：通信/测量/控制/显示/其他
- 简述：[一句话描述功能]

---
请确认/修改这个拆分结果。
输入 `继续` 确认，或补充/修改子系统。
```

**拆分维度**:

| 类型 | 说明 | 示例 |
|------|------|------|
| 通信 | 数据交互 | CAN/UART/I2C/SPI |
| 测量 | 信号采集 | ADC/PWM测量/编码器 |
| 控制 | 输出驱动 | GPIO/PWM/继电器 |
| 显示 | 人机交互 | LED/LCD/串口打印 |
| 状态机 | 流程管理 | 多状态切换 |
| 存储 | 数据管理 | Flash/EEPROM |

---

## 阶段2: 需求确认

**目标**: 对每个子功能确认详细参数

**子功能类型对应问题**:

#### 通信类
```
### [子系统名称] - 需求确认

**通信对象**: （连接什么设备）
**通信协议**: CAN / UART / I2C / SPI / 其他
**通信参数**:
  - 波特率/速率：
  - 地址/ID：
  - 帧格式：
**数据内容**:
  - 发送什么：
  - 接收什么：
**实时性要求**: 实时 / 非实时 / 有deadline
```

#### 测量类
```
### [子系统名称] - 需求确认

**测量对象**: （测什么信号）
**测量精度**: （精度要求）
**测量方式**: 1.轮询  2.中断  3.DMA
**数据处理**: 1.实时计算  2.存储后处理  3.直接输出
**输出格式**: （测量结果怎么用）
```

#### 控制类
```
### [子系统名称] - 需求确认

**控制目标**: （控制什么）
**控制方式**: 1.GPIO开关  2.PWM输出  3.DAC输出
**参数**: 频率/占空比、通道
**触发方式**: 1.定时触发  2.外部触发  3.软件触发
```

---

## 阶段3: 硬件对齐

**目标**: 对涉及硬件的子功能，确认引脚和硬件配置

**输出格式**:

```
### [子系统名称] - 硬件对齐

**已有硬件**:
- [列出已有的相关外设和引脚]

**新增硬件需求**:
1. **IO类型**: 1.数字输入  2.数字输出  3.模拟输入  4.通信接口
2. **引脚分配**: _________
3. **电平**: 1.3.3V  2.5V  3.其他
4. **复用功能**: （如有）
5. **中断需求**: 1.不需要  2.边沿触发  3.电平触发
6. **DMA需求**: 1.不需要  2.需要（用于什么）
7. **芯片手册**: 1.有  2.无
```

---

## 阶段4: 头脑风暴

**目标**: 讨论技术难点和解决方案

**头脑风暴内容**:

| 类型 | 内容 |
|------|------|
| 时序问题 | 中断延迟、响应时间 |
| 资源冲突 | 外设复用、优先级 |
| 可靠性 | 错误处理、恢复机制 |
| 性能 | 实时性优化、算法效率 |
| 兼容性 | 协议一致性、版本兼容 |

**输出格式**:

```
## 头脑风暴

基于前面的讨论，以下是识别的技术难点和解决方案：

### 难点1: [描述]
**分析**: [AI分析原因]
**可能方案**:
  A. [方案A] - 优点/缺点
  B. [方案B] - 优点/缺点
**建议**: [AI推荐]
**确认**: [请用户选择或补充]
```

---

## 阶段5: 子流程拆分

**目标**: 将大需求拆分为可独立开发的子功能

**输出格式**:

```
## 子流程拆分

基于讨论结果，建议拆分为以下子流程：

### S[X]-A: [子功能A名称]
- **所属子系统**: [哪个]
- **开发内容**: 1.[任务1]  2.[任务2]
- **前置条件**: 无
- **验证方式**: [如何验证]
- **优先级**: 1

### S[X]-B: [子功能B名称]
- **所属子系统**: [哪个]
- **开发内容**: 1.[任务1]
- **前置条件**: S[X]-A 完成
- **验证方式**: [如何验证]
- **优先级**: 2

---
请确认：
1. 拆分是否合理？
2. 优先级是否正确？
3. 验证方式是否可行？
```

## 状态管理

**状态文件**: `.emv2/discussion/<需求ID>/status.json`

```json
{
  "discussion_id": "<需求ID>",
  "created_at": "<创建时间>",
  "current_phase": 1,
  "phases": {
    "split": "completed",
    "requirements": "in_progress",
    "hardware": "pending",
    "brainstorm": "pending",
    "milestones": "pending"
  },
  "subsystems": ["子系统A", "子系统B"],
  "current_subsystem": 0
}
```

**状态转换**:

| 用户输入 | AI响应 |
|---------|--------|
| `继续` | 保存当前结果 → 进入下一项/下一阶段 |
| `跳过` | 跳过当前子功能 → 进入下一个 |
| `退出` | 保存进度 → 退出讨论 |
| `继续讨论` | 从上次位置继续 |
