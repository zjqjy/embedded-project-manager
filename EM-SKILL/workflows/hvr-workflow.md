# HVR工作流细则

## 核心流程

```
用户提交验证结果
       ↓
   ┌───┴───┐
   ↓       ↓
 失败     通过
   ↓       ↓
自动创建  更新状态
问题记录   ↓
   ↓    自动归档
进入讨论   (如有闭环问题)
   ↓
分析问题
   ↓
记录解决方案
   ↓
问题状态→进行中
   ↓
用户验证解决
   ↓
闭环记录
```

## 1. 失败触发机制

**触发条件**: 用户输入 `/em result <步骤>-失败`

**AI行为**:
1. 解析失败信息
2. **先更新 HVR 文件**（最重要！）
3. 自动创建问题记录
4. 触发问题讨论流程

## 2. 问题讨论流程

**触发**: 失败记录创建后自动进入

**讨论目标**:
1. 分析问题根因
2. 与用户协作确定解决方案
3. 记录讨论结论

**讨论模板**:

```
## 问题分析

**失败现象**: [从HVR读取]
**相关HVR**: [.emv2/checkpoints/HVR-S<N>-<序号>.md]

━━━━━━━━━━━━━━━━━━━━━━━━━━
请描述：
1. 失败时观察到什么现象？
2. 失败前做了哪些操作？
3. 是否有关键日志或数据？
━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**AI引导分析**:

```
基于描述，我分析可能的原因：

A. **最可能的原因**:
   - [原因1]
   - [原因2]

B. **建议的排查方向**:
   1. 检查[内容1]
   2. 检查[内容2]

请确认或补充...
```

**解决方案记录**:

```
## 解决方案

### 根因分析
[分析结果]

### 解决措施
1. [措施1]
2. [措施2]

### 预期结果
[预期结果]

请确认后开始实施。
```

## 3. 闭环记录

**触发**: 问题解决后，用户提交 `/em result <步骤>-通过` 且原问题已解决

**闭环流程**:

```
1. 更新 problem-log.md 中问题状态 → resolved
2. 记录解决过程
3. 自动归档已闭环记录
```

**闭环记录格式**:

```
### 闭环记录

- **解决日期**: [DATE]
- **解决方案**: [解决方案摘要]
- **验证方式**: [如何验证问题已解决]
- **证据**: [日志/截图等]
```

## 4. problem-log.md 模板

```markdown

- **状态**: open | in_progress | resolved

{{AI和用户共同分析的内容}}

### 闭环记录
- **解决日期**: -
- **解决方案**: -
- **验证方式**: -
- **证据**: -

---

## 历史归档
<!-- 已闭环问题归档索引 -->
```

## 5. HVR文件增强

**HVR-SX-XXX.md 内容**:

```markdown
## 人工验证请求 [HVR-S<N>-<序号>]

**验证类型**: 硬件功能测试
**所属步骤**: [步骤名称]
**前置条件**: 已编译通过
**创建时间**: {{DATETIME}}

### 操作清单（人工执行）
- [ ] 1. 烧录固件
- [ ] 2. 硬件连接
- [ ] 3. 上电观察
- [ ] 4. 记录结果

### 预期结果
- [预期输出]

### 实际结果（人工填写）
[待用户填写]

### 结论
□ 通过  □ 失败  □ 部分通过

---

### 失败时记录
[失败时自动填充]
- **失败时间**: {{DATETIME}}
- **失败现象**: {{用户描述}}
- **相关日志**: {{如有}}

---

**提交命令**: /em result S<N>-通过 或 /em result S<N>-失败-[描述]
```

## 6. 工作流状态

| 用户输入 | AI响应 |
|---------|--------|
| `/em result s2-失败-串口无输出` | 自动创建问题记录 → 进入问题讨论 |
| `/em result s2-通过` | 更新状态 → 提示下一步 |
| 问题讨论中 | AI引导分析 → 用户确认 → 记录解决方案 |
| `追问 xxx` | 在讨论中回答用户问题 |

## 关键提醒

⚠️ **失败时必须按顺序执行：**
1. 先更新 HVR 文件
2. 再创建 problem-log.md
3. 最后进入讨论模式
