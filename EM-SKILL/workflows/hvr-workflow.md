# HVR工作流细则（人-AI协作版）

## 核心定位

**HVR是AI维护、人类确认的工作区**

- **人类只做**：执行验证操作、口头描述观察
- **AI负责**：记录到HVR文件、分析根因、给出建议、维护文档

## 核心流程

```
┌─────────────────────────────────────────────────────────────┐
│  1. 准备阶段                                               │
├─────────────────────────────────────────────────────────────┤
│  用户: /em verify s3                                      │
│       ↓                                                   │
│  AI: 🔄 状态: 🚧开发中→🔄验证中                           │
│  AI: 生成HVR                                              │
│  AI: 调用 EM-SKILL 工具:                                   │
│      - 需要编译 → 调用 build-keil                          │
│      - 需要烧录 → 调用 flash-openocd                       │
│      - 需要串口 → 启动 serial-mcp/serial-monitor           │
│  AI: 记录工具执行结果到 HVR                                │
└─────────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────────┐
│  2. 验证阶段 ←── /em result 在这里                         │
├─────────────────────────────────────────────────────────────┤
│  用户: 执行验证 → 观察现象 → 提交结果                      │
│       ↓                                                   │
│  用户: /em result s3-通过 或 /em result s3-失败-描述      │
└─────────────────────────────────────────────────────────────┘
                          ↓
          ┌─────────────────┴─────────────────┐
          ↓                                   ↓
    【通过】                              【失败】
          ↓                                   ↓
    🔄 状态:🔄验证中→✅完成             🔄 状态:🔄验证中→🔁返工中
    AI更新HVR                              AI更新HVR
    🔄 自动推进到下一子步骤                AI: [读取MCP日志]
    更新project-spec                          + 人类补充观察
    提示下一步                                  ↓
                                        AI + 人类 共同分析
                                                  ↓
                                        AI + 人类 共同修改
                                                  ↓
                                      🔄 状态:🔁返工中→🔄验证中
                                           重新验证
```

## 两种情况

### 情况A：验证通过
```
用户: /em result s3-通过
AI: 🔄 更新状态: ✅ 完成 → 自动推进到下一子步骤
AI: 更新HVR → 更新project-spec → 提示下一步
```

### 情况B：验证失败
```
用户: /em result s3-失败-LED闪红
AI: 🔄 更新状态: 🔁 返工中 → 更新 Meta
AI: [读取MCP日志] → 分析原因
AI: "可能原因1/2/3，请确认"
用户: "确认是原因2"
AI + 人类 共同分析 → 确定修改方案
AI + 人类 一起修改代码
用户: /em verify s3  → 🔄 状态: 🔁返工中→🔄验证中
重新验证 → /em result s3-通过
```

## 1. 失败触发机制

**触发条件**: 用户输入 `/em result <步骤>-失败`

**AI行为**:
1. 解析失败信息
2. **先更新 HVR 文件**（最重要！）
3. 自动创建问题记录
4. 触发问题讨论流程

## 2. 问题讨论流程

**触发**: 失败记录创建后自动进入

**讨论目标**:
1. 分析问题根因
2. 与用户协作确定解决方案
3. 记录讨论结论

**讨论模板**:

```
## 问题分析

**失败现象**: [从HVR读取]
**相关HVR**: [.emv2/checkpoints/HVR-S<N>-<序号>.md]

━━━━━━━━━━━━━━━━━━━━━━━━━━
请描述：
1. 失败时观察到什么现象？
2. 失败前做了哪些操作？
3. 是否有关键日志或数据？
━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**AI引导分析**:

```
基于描述，我分析可能的原因：

A. **最可能的原因**:
   - [原因1]
   - [原因2]

B. **建议的排查方向**:
   1. 检查[内容1]
   2. 检查[内容2]

请确认或补充...
```

**解决方案记录**:

```
## 解决方案

### 根因分析
[分析结果]

### 解决措施
1. [措施1]
2. [措施2]

### 预期结果
[预期结果]

请确认后开始实施。
```

## 3. 闭环记录

**触发**: 问题解决后，用户提交 `/em result <步骤>-通过` 且原问题已解决

**闭环流程**:

```
1. 更新 problem-log.md 中问题状态 → resolved
2. 记录解决过程
3. 自动归档已闭环记录
```

**闭环记录格式**:

```
### 闭环记录

- **解决日期**: [DATE]
- **解决方案**: [解决方案摘要]
- **验证方式**: [如何验证问题已解决]
- **证据**: [日志/截图等]
```

## 6. 工作流状态

| 用户输入 | AI响应 |
|---------|--------|
| `/em result s2-失败-串口无输出` | 自动创建问题记录 → 进入问题讨论 |
| `/em result s2-通过` | 更新状态 → 提示下一步 |
| 问题讨论中 | AI引导分析 → 用户确认 → 记录解决方案 |
| `追问 xxx` | 在讨论中回答用户问题 |

## 关键提醒

⚠️ **失败时必须按顺序执行：**
1. 先更新 HVR 文件
2. 再创建 problem-log.md
3. 最后进入讨论模式
4. 讨论结果必需记录
